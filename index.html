What this tells us about thresholds

M2 missing + active covers 32 of 33 active teams — too broad to use alone, needs a pairing signal. s2_survey_response = 0 on an active team is a meaningful gap indicator. work_log_sum < 600 on active teams warrants a watch tier. How the AI analysis works

From CSV to educator update

This is the prompt your team will give to Claude. Paste it with the CSV attached.

Step 1 — Analysis prompt (copy & paste into Claude)
Prompt · Analytics layer
You are an experience manager at Riipen analyzing weekly cohort data.
The attached CSV is the team activity tracker export.

Analyze the data and produce:

1. A risk table with three tiers:
   RED: flag_count ≥ 2, OR work_log_sum < 500 AND team_state = "active", OR kickoff_id is blank AND team_state = "active"
   YELLOW: flag_count = 1, OR conversation_summary = "slightly_negative", OR m2_submission_count is blank AND team_state = "active" AND m1_submission_count ≥ 1
   GREEN: active teams with no flags and milestone progression on track

   Format: Team Name | Project | Tier | Primary Signal | Suggested Action

2. A cohort snapshot (3–5 sentences):
   - Total active, completed, at-risk
   - Highest-flag teams by name
   - Milestone gap pattern
   - Survey response gaps

Do not hallucinate. Blank cell = no data, not zero.
Step 2 — Narrative prompt (after step 1 output)
Prompt · Educator update
Using the risk analysis above, write a weekly educator update email.

Style:
- Warm but direct, like a colleague briefing you
- Open with 1–2 sentences of overall health
- Include the risk table as "Teams to watch"
- For each RED team: one action sentence starting with "I'd suggest..."
- Close with one positive observation
- Under 250 words total
- No formal boilerplate ("pursuant to", "please be advised")

Do not invent team names or statistics not in the input.

What this replaces

Jen currently spends ~20–40 min per cohort going row by row through the dashboard. This workflow gets to the same output in under 3 minutes: 1 CSV export + 1 prompt paste + 1 click send.

IT clearance boundary

This workflow requires zero IT approval. Everything runs on tools Jen already has (Claude, ChatGPT) and uses only files she manually exports. No API keys. No platform credentials stored anywhere. No automated platform access.

The agentic route (where the AI logs into Riipen directly and pulls data on a schedule) is documented in the final recommendation as a future-state option — with exactly what IT would need to approve.

Feedback prompts for the workflow review

These short prompts are here to help us tune the prompt library, thresholds, and handoff guidance after you review the proposed workflow.

🎯 Threshold calibration

We have proposed thresholds from the data. These questions let Jen validate or adjust them.

  • Must ask · When you see a flag_count of 2 or more on an active team, what's your first action — do you go into the platform immediately, or does it need a second signal before you escalate?
    Why it matters: This sets the Red vs Watch boundary. Right now we're using flag ≥ 2 as Red, but if Jen acts only on flag + another signal, we need to tighten it.
    +
  • Must ask · Almost every active team in Export 3 is missing M2. Is that expected at this stage of the program, or is a missing M2 actually a signal worth flagging?
    Why it matters: 32 of 33 active teams have no M2 submission. If M2 isn't due yet, this is noise. We need to know the program timeline before we use milestone gaps as a signal.
    +
  • Good to ask · What's the typical work_log_sum for a healthy team at this stage? We see a range from 330 to 8,325 in Export 1 — what floor would you consider low enough to flag?
    Why it matters: Work log is our best proxy for effort, but without a program-specific baseline we can't set a meaningful threshold.
    +
  • Good to ask · If a team has zero survey responses on S2 but no flags, do you consider that a risk or just an incomplete data point?
    Why it matters: 14 teams in Export 3 have zero S2 responses. We need to know whether to flag that alone or only in combination with another signal.
    +

📧 Educator update format

We've built on the example updates Jen sent. These questions lock in the template before we finalize prompts.

  • Must ask · The three update examples you sent have different formats — one is a student-by-student breakdown, one is narrative only, one is narrative plus action items. Which of those comes closest to what you'd want every week?
    Why it matters: We need one canonical template to train our prompts on. Right now we're targeting the hybrid (table + narrative), but Jen should confirm.
    +
  • Must ask · When you write the "Riipen action taken" column in Example 1 — is that something you want the AI to suggest, or is that always written by you because it involves platform-level actions the educator can't take?
    Why it matters: Example 1 has a very specific format with Riipen-side actions. If that's Jen's internal record (not the educator update), we're building a different output than we think.
    +
  • Good to ask · Do different educators want different formats, or do you send the same structure to everyone?
    Why it matters: If educators have preferences, the system needs to support multiple templates. If it's one standard, we build one template and we're done.
    +
  • If time · How often do you mention specific team names in the educator update vs. keeping it cohort-level?
    Why it matters: Example 2 uses [Team X] placeholders. Example 1 uses student names. Knowing the norm helps us calibrate how specific the AI output should be.
    +

🔁 Workflow & scale

Process questions that affect how we design the handoff guide.

  • Must ask · How many active experiences are you monitoring right now, and how many CSV files does that mean you're exporting per week?
    Why it matters: If it's 5 experiences, the manual load is manageable. If it's 25+, we need to think about batching or a multi-file prompt strategy.
    +
  • Good to ask · What day and time do you typically send the educator update — and does the export need to happen right before, or is a 24-hour-old snapshot fine?
    Why it matters: Determines whether we need to account for data freshness in the prompt, and informs the weekly run cadence in the handoff guide.
    +
  • Good to ask · Is it always just you running this, or would another experience manager on your team also be using this workflow?
    Why it matters: If multiple people use it, the handoff guide needs to be more thorough and the prompts need to work without tribal knowledge about the data structure.
    +
  • If time · You mentioned a possible end-of-program summary update — would that use the same CSV format, or does it pull from different data?
    Why it matters: If the end-of-program update is a separate use case, we may want to build a second prompt variant for it in Week 5.
    +
